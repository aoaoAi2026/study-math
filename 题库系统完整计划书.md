# 小学数学知识宝典 — 题库系统完整计划书

> 版本: v1.0 | 状态: 待确认 | 适用: 1~6 年级基础数学 & 奥数

---

## 一、项目现状诊断

### 1.1 当前已有的东西

| 模块 | 文件路径 | 现状说明 |
|------|---------|---------|
| 📚 知识讲解内容 | `public/content/grade-X/basic/*.md`<br>`public/content/grade-X/olympiad/*.md` | 每个知识点 1 个 Markdown 文件<br>含故事卡/概念卡/例题卡/变式卡/错卡/亲子卡 |
| ❓ 题目数据 | `src/stores/exerciseData.ts` | ⚠️ **所有题目挤在一个 TypeScript 文件里**<br>当前每个知识点约 2~3 道题<br>仅 50% 知识点有题目<br>题型仅限 choice（选择）与 fill（填空） |
| 🔧 练习引擎 | `src/services/exerciseEngine.ts` | 有 IRT（项目反应理论）能力估计 + 智能抽题 + L1-L4 错误分类 |
| 🎯 练习 UI | `src/components/practice/QuizEngine.vue`<br>`src/components/practice/ChoiceQuestion.vue`<br>`src/components/practice/FillQuestion.vue` | 有基础答题 UI，但缺少 "做课后测试" 的入口与流程 |
| 📝 错题本 | `src/features/profile/WrongBookPage.vue` | 已有 UI 框架，但无真实错题持久化 |

### 1.2 核心问题（用户反馈的痛点）

1. ❌ **没有独立的题库文件** — 所有题目挤在一个 `exerciseData.ts` 里，无法独立编辑、扩展、审查
2. ❌ **题量太少** — 每个知识点仅 2~3 道，不够做"课后测试"（至少需 5~10 道）
3. ❌ **难度梯度不足** — 缺少从 "入门 → 基础 → 进阶 → 竞赛" 的分层
4. ❌ **缺少题型多样性** — 无应用题/列式题/几何画图题/逻辑推理题/火柴棒拼图题
5. ❌ **没有真题来源** — 当前题目是手写的简单题，没有像杯赛/小升初/课本习题这类真实题目
6. ❌ **知识点 ID 与内容文件未对齐** — 题目中出现的 ID（如 `g1-sum-multiple`、`chicken-rabbit`）在 content 目录中可能不存在对应讲解
7. ❌ **课后测试流程缺失** — 用户在学完一个知识点后，没有 "开始测试" 的按钮和完整流程

## 二、重构目标

| 目标 | 量化指标 |
|------|---------|
| 每个知识点 1 个独立题库文件 | 覆盖 **~127 个** 知识点专题 |
| 每个题库文件至少 15 道题 | 含 5 档难度（入门 3 / 基础 4 / 进阶 4 / 拔高 2 / 竞赛 2） |
| 5 种题型 | choice / fill / input / matchstick / shape-match |
| 真题来源标记 | 每道题有 `source` 字段（教材/杯赛/考试年份） |
| 智能抽题 | 课后测试 = 按知识点过滤 + 按难度随机 + 按能力权重挑题 |
| 题型完整化 | choice（4选项）/ fill（填空）/ input（列式答题）/ matchstick（火柴棒等式）/ shape-match（图形几何题） |
| 错题持久化 | localStorage 存答题记录，按 L1-L4 分类，错题本可重做 |

**总题目规模目标: 127 知识点 × 15 题 ≈ 1,900 道题**

### 3.3 完整题目类型定义

```typescript
type QuestionType = 'choice' | 'fill' | 'input' | 'matchstick' | 'shape-match'

type Difficulty = 1 | 2 | 3 | 4 | 5

interface Exercise {
  id: string                      // 唯一ID，格式: g{年级}-{专题缩写}-{序号}
  type: QuestionType              // 题型
  difficulty: Difficulty          // 难度(1-5)
  difficultyLabel: string         // 难度标签(入门/基础/进阶/拔高/竞赛)
  stem: string                    // 题干(支持Markdown简单格式)
  image?: string                  // 图像资源ID(可选)
  options?: string[]              // 选择题选项(choice题型必填)
  answer: string                  // 标准答案
  acceptableAnswers?: string[]    // 可接受答案(用于填空题/输入题)
  solution: string[]              // 解析步骤
  hints: string[]                 // 提示(可逐步展开)
  commonMistakes: CommonMistake[] // 常见错误及原因
  source: QuestionSource          // 题目来源（真题标记）
  knowledgePoints: string[]       // 关联知识点
  estimatedTime: number           // 预计用时(秒)
  irtParams: { a: number; b: number; c: number }  // IRT参数
  tags: string[]                  // 标签/关键词
}

interface CommonMistake {
  mistake: string                 // 错误答案/错误行为描述
  reason: string                  // 错误原因分析
  correction: string              // 正确思路提示
  errorLayer: 'L1' | 'L2' | 'L3' | 'L4'  // 错误层级
}

interface QuestionSource {
  type: 'textbook' | 'competition' | 'exam' | 'original'
  name: string                    // 来源名称(如"希望杯"/"人教版"/"小升初")
  year?: number                   // 年份
  original?: string               // 原题编号/出处
}
```

### 3.4 难度分级标准

| 难度 | 标签 | 含义 | IRT b 值范围 | 年级水平对应 |
|------|------|------|---------------|-------------|
| 1 | 入门 | 直接套用公式/例题，一步就能出答案 | -1.0 ~ -0.5 | 低年级 / 刚学完 |
| 2 | 基础 | 需要理解概念，2~3 步简单运算 | -0.5 ~ 0.0 | 本年级基础题 |
| 3 | 进阶 | 需要 3~5 步，可能有小陷阱，需要一定的推理 | 0.0 ~ 0.6 | 本年级中档题 |
| 4 | 拔高 | 需要 5+ 步，含技巧/公式变形，综合运用 | 0.6 ~ 1.2 | 高年级水平 |
| 5 | 竞赛 | 真正的奥数题，需要巧思/模型/组合思维 | 1.2 ~ 2.0 | 杯赛难题 |

## 五、课后测试抽题算法

### 5.1 抽题流程图

```
用户点击 "开始课后测试"
    │
    ▼
[1] 获取当前知识点 topicId
    │
    ▼
[2] 读取题库文件 question-bank/{topicId}.json
    ├── 如果文件不存在 → 使用通用抽题（从总题库按 ID 过滤）
    │
    ▼
[3] 读取用户能力数据（localStorage 中保存的能力值）
    ├── ability = 用户对该知识点的 IRT 能力估计（-2 ~ +2）
    │
    ▼
[4] 调用 pickExercises(pool, ability, opts)
    │
    ├── 参数:
    │   ├── knowledgePoints = [topicId]
    │   ├── count = 5 ~ 8 (根据难度自适应)
    │   ├── minDiff = max(1, floor(ability + 1))
    │   └── maxDiff = min(5, ceil(ability + 3))
    │
    ├── 算法:
    │   │
    │   ├─ 按知识点过滤 → 候选池 A
    │   ├─ 按难度区间过滤 → 候选池 B
    │   ├─ 对每道题计算 p = IRT(ability, a, b, c)
    │   ├─ 优先选 |p - 0.6| 最小的（约 60% 正确率，有挑战但不失信心）
    │   ├─ 确保题目类型多样性（至少 2 种题型）
    │   └─ 确保难度覆盖（至少 1 道基础题 + 1 道进阶题）
    │
    ▼
[5] 生成测试 session
    ├── sessionId
    ├── exercises[...题目列表]
    ├── startTime
    └── expectedDuration
    │
    ▼
[6] 逐题显示在 QuizEngine 中
    └── 每题答题后即时判断对错 → 更新用户能力值
    │
    ▼
[7] 测试结束后
    ├── 显示正确率
    ├── 显示错题列表（可展开看解析）
    ├── 错题写入 WrongBook（按 L1-L4 分类）
    ├── 建议下一步（再练一轮/升级难度/回顾知识点）
    └── 更新学习记录 (LearningStore)
```

## 六、题目内容编写指南（给出题人看）

### 6.1 编写流程

1. **打开** 对应的 Markdown 讲解文件（理解知识点核心概念）
2. **阅读** 例题卡 / 变式卡，理解该知识点的典型解法
3. **按 5 档难度** 编写题目:
   - 入门(3题): 直接照搬例题，替换数字
   - 基础(4题): 小变形，考察概念理解
   - 进阶(4题): 多步骤/含陷阱/需画图
   - 拔高(2题): 综合题/需公式变形或推理
   - 竞赛(2题): 真正的奥数杯赛题或改编题
4. **每题必须写** stem / answer / solution[] / hints[] / commonMistakes[] / source
5. **保存** 到 `public/content/question-bank/gX/{topic-id}.json`

### 6.2 难度梯度示例（以 "和倍问题" 为例）

| 难度 | 题干示例 | 考察点 |
|------|---------|--------|
| 入门(1) | 两数之和为 10，大数是小数的 4 倍。求小数。 | 直接套用公式: 小数 = 和 ÷ (倍数+1) |
| 基础(2) | 苹果和梨共 20 个，苹果是梨的 3 倍。苹果几个？ | 把"情境"转化为和倍关系 |
| 进阶(3) | 甲是乙的 2 倍，丙是甲的 3 倍，三数之和为 81。求乙。 | 三个量的和倍，需建立统一段数 |
| 拔高(4) | 书架上下层共 120 本书，从上层拿 15 本到下层后上层仍是下层的 2 倍。原上下层各几本？ | 和不变 + 倍数变化 + 倒推 |
| 竞赛(5) | 甲乙丙三人分糖果，甲分得的是乙丙之和的 2/3，乙分得的是甲丙之和的 3/5，丙分得 7 颗。三人共分几颗糖？ | 需把比例转化为和倍模型，真正奥数水平 |

### 6.3 真题来源建议

| 年级 | 来源类型 | 具体来源 | 难度标签 |
|------|---------|---------|---------|
| 1-2年级 | 教材课后题 | 人教版/苏教版/北师大版 课本习题 | 入门/基础 |
| 1-3年级 | 低年级竞赛 | "数学大王"邀请赛 / "希望杯"低年级组 | 基础/进阶 |
| 3-4年级 | 典型奥数 | "华罗庚金杯" 初赛题 / "希望杯"初赛 | 进阶/拔高 |
| 4-5年级 | 中档竞赛 | "华杯赛"复赛题 / "走美杯" / "希望杯"复赛 | 拔高 |
| 5-6年级 | 高级竞赛 | 华杯赛决赛题 / 迎春杯 / 学而思杯 | 竞赛 |
| 6年级 | 小升初 | 历年民办初中入学测试题 | 拔高/竞赛 |

**注意**: 由于版权原因，我们不应直接复制原题，建议**改编**（改数字/改情境），并在 source 中标记 "改编自 xxx"。

### 6.4 题干格式规范

- 使用简洁中文，避免口语化
- 数字与单位之间加空格: "5 cm" 而非 "5cm"
- 数学表达式保持可读: "1/3" 表示分数，"a²" 表示平方
- 含图形的题目标记 `image` 字段（如 `"image": "shape-triangle-area"`）
- 题目开头应有清晰的情境描述或条件说明

- [ ] 如果是填空/输入题，acceptableAnswers 涵盖合理写法

### 8.2 JSON 格式检查

每个题库文件应通过以下验证:

```bash
# 用 Node.js 验证 JSON 合法性（写个脚本）
node scripts/validate-question-bank.js
```

检查项目:
- JSON 语法正确
- 必填字段存在
- answer 与 solution 一致
- exercises 数组非空
- 每题 difficulty 在 1-5 之间
- 选择题有且仅有 4 个 options
- answer 指向正确的 option index（choice）或填写了答案文本（fill/input）

### 8.3 自动测试清单

- [ ] **测试加载**: 随机抽取 10 个题库文件，验证能完整读取
- [ ] **测试答题**: 随机用 2 种难度做一轮 5 题的课后测试

- [ ] **测试空数据容错**: 删除一个题库文件，验证 fallback 到旧数据不崩溃

## 九、风险与应对

| 风险 | 影响 | 概率 | 应对 |
|------|------|------|------|
| 题库量太大，人力跟不上 | 上线延迟 | 中 | 先做"基础版" 600 题覆盖所有专题的基础题，后续分批补齐 |

| JSON 手写易错 | 解析失败 | 低 | 提供 JSON 模板 + 验证脚本，或改用工具化出题界面（后续迭代） |
| 高年级难度控制不准 | 孩子受挫 | 中 | 先保守出题（偏易），用真实答题数据动态调高/调低 |
| 题库文件太多导致加载慢 | UI 卡顿 | 低 | 按专题懒加载（只在用户点进某专题时才读该题库），无需一次加载所有 JSON |
| 家长要求"看孩子做了什么" | 功能需求 | 高 | 在家长看板页增加"学习轨迹"和"错题分析"模块 |
